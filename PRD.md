# PRD — 경쟁사 아침 브리핑 도구

> 매일 아침 경쟁사 동향을 정리해 브리핑을 만들고, '배포'하면 GitHub Pages에 카드로 쌓이는 공개 아카이브를 생성하는 도구.

- **작성일**: 2026-07-14
- **오너**: minsu ban (halfminsu)
- **한 줄 정의**: 경쟁사 이름 + 키워드를 넣으면 그날 챙길 항목을 자동 수집·요약한 브리핑을 만들고, 클릭 한 번으로 공개 아카이브에 배포한다.

## 확정된 방향 (결정 사항)
| 항목 | 결정 |
|------|------|
| 데이터 수집 | **자동 수집** — 웹/뉴스 API 호출로 그날 항목을 끌어옴 |
| 배포 방식 | **GitHub Pages + 정적 파일** — 데이터 커밋/푸시 → 정적 사이트 공개 |
| 로컬 도구 형태 | **로컬 웹앱** — localhost 브라우저에서 입력·미리보기·배포 |

## 전체 구조 (두 부분)
```
[내 PC]                              [GitHub]
┌─────────────────────────┐         ┌──────────────────────────────┐
│  Part A                 │  push   │  Part B                      │
│  브리핑 생성 도구        │ ──────▶ │  브리핑 아카이브 (Pages)      │
│  (로컬 웹앱)            │         │  카드 인덱스 + 개별 상세 페이지 │
│  수집→요약→미리보기→배포 │         │  공개 URL, 정적 파일          │
└─────────────────────────┘         └──────────────────────────────┘
        └── 브리핑 데이터(JSON/MD) 를 저장소에 커밋 ──┘
```

- **데이터가 진실의 원천(source of truth)**: Part A는 브리핑을 `briefings/YYYY-MM-DD.json`(+ 본문 MD)로 저장하고 git push. Part B는 그 파일들을 읽어 카드/상세 페이지로 렌더.
- 두 부분은 **브리핑 데이터 스키마**로만 연결된다(느슨한 결합).

# Part A — 브리핑 생성 도구 (로컬 웹앱)

## 목적
매일 아침, 경쟁사·키워드를 입력하면 자동으로 동향을 수집·정리한 브리핑 초안을 만들고, 사람이 검토/편집한 뒤 아카이브로 배포한다.

## 사용자 & 시나리오
- 사용자: 나(1인). 매일 아침 5~10분 사용.
- 시나리오: 브라우저에서 로컬 도구 열기 → 경쟁사/키워드 입력 → "브리핑 생성" → 자동 수집·요약 결과 검토·편집 → "배포" 클릭 → 공개 아카이브에 반영.

## 핵심 기능 (요구사항)
| ID | 기능 | 설명 | 우선순위 |
|----|------|------|:---:|
| A1 | 입력 폼 | 경쟁사 이름(복수), 키워드(복수), 기간(기본=오늘) 입력 | P0 |
| A2 | 자동 수집 | 뉴스/웹 검색 API로 경쟁사×키워드 관련 최신 항목 수집 | P0 |
| A3 | 정리·요약 | 수집 항목을 중복 제거·그룹핑하고 핵심 요약 생성 | P0 |
| A4 | 브리핑 미리보기 | 날짜·제목·핵심 요약·항목 리스트(출처 링크 포함) 렌더 | P0 |
| A5 | 편집 | 제목/요약/항목을 손으로 다듬기 (불필요 항목 삭제) | P0 |
| A6 | 저장 | 브리핑을 `briefings/YYYY-MM-DD.{json,md}` 로 저장 | P0 |
| A7 | 배포 | 저장 파일을 git add/commit/push (→ Pages 반영) | P0 |
| A8 | 프리셋 | 자주 쓰는 경쟁사/키워드 세트 저장·불러오기 | P1 |
| A9 | 히스토리 | 과거 브리핑 목록 로컬에서 열람·재편집 | P1 |
| A10 | 중복/변화 감지 | 어제 대비 새로 뜬 항목 하이라이트 | P2 |
| A11 | **중요도 표시** | 각 항목에 **상·중·하** 중요도 지정(자동 초안 + 수동 조정) | P0 |
| A12 | 중요도 정렬/필터 | 미리보기에서 중요도순 정렬, '중 이상만 보기' 필터 | P1 |

## 브리핑 데이터 스키마 (A·B 공통 계약)
```jsonc
// briefings/2026-07-14.json
{
  "date": "2026-07-14",
  "title": "경쟁사 아침 브리핑 — 7월 14일",
  "summary": "오늘 핵심 3줄 요약 …",       // 카드에 노출
  "top_importance": "high",              // 그날 항목 중 최고 중요도 (카드 뱃지용): high|medium|low
  "competitors": ["A사", "B사"],
  "keywords": ["가격인하", "신규채용", "신제품"],
  "items": [
    {
      "competitor": "A사",
      "keyword": "신제품",
      "headline": "A사, 신형 X 출시 발표",
      "note": "왜 중요한지 한 줄",
      "importance": "high",              // 상=high · 중=medium · 하=low
      "url": "https://…",
      "source": "출처명",
      "published_at": "2026-07-14T08:12:00+09:00",
      "is_new": true
    }
  ],
  "generated_at": "2026-07-14T09:00:00+09:00"
}
```
- 본문 상세는 `briefings/2026-07-14.md`(사람이 읽기 좋은 형태)로 병행 저장 가능.
- **날짜 = 파일명 = 카드 식별자**. 하루 1개 브리핑 기준(초과 시 `-2` 접미사 규칙).

## 처리 흐름
1. 입력(A1) → 2. 수집(A2: API 호출, 경쟁사×키워드 쿼리) → 3. 정리·요약(A3) → 4. 미리보기(A4) → 5. 편집(A5) → 6. 저장(A6) → 7. 배포(A7: 커밋/푸시).

## 기술 방향 (제안)
- 스택: 로컬 정적 프런트(HTML/JS) + 얇은 로컬 서버(Node/Python) — API 키를 브라우저에 노출하지 않기 위해 서버가 외부 API 호출과 git 명령을 대행.
- 외부 API: 뉴스/웹 검색 API 1개 선택(예: 뉴스 검색 API). 요약은 규칙 기반 또는 선택적으로 LLM.
- Git 연동: 이미 인증된 `gh`/HTTPS 사용(현재 환경에 준비됨).

## 비기능 요구
- API 키 등 비밀값은 로컬 `.env`에만 두고 **저장소에 커밋 금지**(`.gitignore`).
- 배포는 되돌릴 수 있어야 함(잘못 배포 시 해당 파일 삭제 커밋으로 롤백).
- 외부 API 실패 시에도 수동 입력으로 브리핑 작성 가능(폴백).

## 중요도(상·중·하) 규칙 — A·B 공통
- 값: `high`(상) · `medium`(중) · `low`(하). 각 **항목(item)** 에 하나씩 부여.
- 자동 초안: 수집 단계에서 기본 규칙으로 중요도 초안 제시(예: 지정 경쟁사 정면 언급·다수 출처 노출·특정 키워드[가격/M&A 등] 매칭 시 상향). 최종값은 사람이 조정(A11).
- 카드 뱃지용 `top_importance` = 그날 항목 중 최고 중요도.

## Part A 완료 기준(수용 조건)
- [ ] 경쟁사/키워드 입력 → 5초 내 자동 수집 결과가 미리보기에 뜬다.
- [ ] 각 항목에 중요도 초안(상·중·하)이 붙고, 드롭다운 등으로 손쉽게 바꿀 수 있다.
- [ ] 편집 후 저장하면 스키마(`importance`, `top_importance` 포함)에 맞는 JSON/MD가 `briefings/`에 생긴다.
- [ ] "배포" 클릭 한 번으로 커밋·푸시가 완료되고, 몇 분 뒤 공개 페이지에 카드가 추가된다.

# Part B — 브리핑 아카이브 (GitHub Pages)

## 목적
배포된 브리핑들을 **공개 URL에서 카드 목록으로 열람**하고, 카드 클릭 시 그날 상세 브리핑을 볼 수 있는 정적 사이트.

## 페이지 구성
| ID | 화면 | 내용 |
|----|------|------|
| B1 | 인덱스(카드 목록) | 브리핑들을 최신순 카드로 나열. 카드 = 날짜 · 제목 · 핵심 요약 |
| B2 | 상세 페이지 | 카드 클릭 시 그날 브리핑 전체(항목·출처 링크·노트) |
| B3 | 헤더/소개 | 사이트 제목, 용도 한 줄, 최신 업데이트 날짜 |
| B4 | 중요도 필터/정렬 | 인덱스에서 중요도(상·중·하)로 필터·정렬 |

## 카드 사양 (B1)
- 표시: **날짜**(굵게), **제목**, **핵심 요약**(2~3줄, 말줄임), 경쟁사 태그, **중요도 뱃지(상·중·하)**.
- 중요도 뱃지: 카드의 `top_importance`를 색으로 구분(상=빨강/강조, 중=노랑, 하=회색). 색만이 아니라 텍스트(상/중/하)도 함께 표기(색약 대비).
- 정렬: 날짜 내림차순(최신이 위). (선택) 중요도 우선 정렬 토글.
- 필터: **'중 이상만 보기' 등 중요도 필터** (B4 참고).
- 동작: 카드 클릭 → 해당 날짜 상세(B2)로 이동.
- 반응형: 데스크톱 다열 그리드 / 모바일 1열.

## 상세 페이지 사양 (B2)
- 상단: 날짜·제목·핵심 요약.
- 본문: 항목 리스트(경쟁사·키워드·헤드라인·노트·출처 링크), 새 항목(`is_new`) 배지, **항목별 중요도 뱃지(상·중·하)**.
- 정렬: 항목을 **중요도순(상→하)** 으로 정렬해 표시.
- 하단: 목록으로 돌아가기 링크.

## 데이터 → 렌더 방식
- 소스: 저장소의 `briefings/*.json`.
- 인덱스 생성: 배포 시 `briefings/index.json`(카드용 목록: date/title/summary/competitors/**top_importance**)을 만들어 두면 인덱스 페이지가 빠르게 렌더. (생성 주체는 Part A의 배포 단계 또는 간단한 빌드 스텝.)
- 정적 파일만 사용하므로 서버 불필요.

## 배포/호스팅
- **GitHub Pages** 로 공개. 저장소 루트 또는 `/docs`에 정적 페이지 배치.
- 브리핑 데이터가 push되면 Pages가 자동으로 최신본을 서빙.
- 공개 URL: `https://halfminsu.github.io/<repo>/`.

## 비기능 요구
- 공개 페이지이므로 **민감/내부 정보 노출 금지** — 브리핑 본문은 공개 가능한 내용만.
- 데이터가 0건이어도 인덱스가 깨지지 않고 "아직 브리핑이 없습니다" 표시.
- 접근성: 카드 키보드 포커스/링크 이동 가능, 명도 대비 확보.

## Part B 완료 기준(수용 조건)
- [ ] 공개 URL에 접속하면 최신순 카드 목록이 보인다.
- [ ] 카드에 날짜·제목·핵심 요약·**중요도 뱃지(상·중·하)** 가 정확히 표시된다.
- [ ] 중요도 필터('중 이상만 보기' 등)가 동작한다.
- [ ] 카드를 누르면 그날 상세 페이지가 열리고, 항목이 **중요도순**으로 출처 링크와 함께 보인다.
- [ ] 브리핑을 새로 배포하면 몇 분 내 카드가 하나 늘어난다.

```

├─ tool/                 # Part A: 로컬 웹앱 (배포 대상 아님)

│  └─ assets/
├─ briefings/            # 공통 데이터 (진실의 원천)
│  ├─ 2026-07-14.json
│  ├─ 2026-07-14.md
│  └─ index.json         #   카드용 목록
├─ .env                  # 비밀값 (gitignore)
└─ .gitignore
```

## 범위 밖 (이번 버전 안 함)
- 다중 사용자/로그인, 댓글·구독, 알림(이메일/슬랙).

- 유료/고급 분석(감성분석, 경쟁사 자동 발굴).

## 열린 질문
1. 자동 수집에 쓸 **뉴스/웹 검색 API**를 무엇으로 할지(무료 한도·언어 지원).
2. 요약을 규칙 기반으로 할지, **LLM(Claude API)** 을 붙일지.
3. 아카이브 저장소를 **공개**로 둘지(브리핑 내용 공개 범위와 직결).
