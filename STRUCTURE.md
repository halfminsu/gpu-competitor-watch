# 구조 정리 — 구현 전 설계

> [PRD.md](PRD.md) 를 바탕으로, 실제로 만들기 전에 화면·데이터·흐름·우선순위를 확정한다.

---

## ① 화면 구성 (총 2개)

| 화면 | 이름 | 어디서 | 하는 일 |
|---|---|---|---|
| **화면 1** | **브리핑 생성 화면** | 내 PC (로컬 웹앱) | 한 화면 안에서 위→아래로: ① 경쟁사·키워드 입력 → ② 자동 수집 결과 미리보기·편집(항목 삭제, **중요도 상·중·하 조정**, 요약 수정) → ③ "배포" 버튼 |
| **화면 2** | **아카이브 화면** | GitHub Pages (공개) | 기본은 **카드 목록**(날짜·제목·요약·중요도 뱃지). 카드를 누르면 같은 화면 개념 안에서 **그날 상세**로 전환(별도 URL/페이지지만 하나의 "아카이브 화면"으로 취급) |

- 화면 1은 단계별 화면 전환 없이 **한 페이지에 다 넣는다** (입력 → 결과 → 배포까지 스크롤/섹션으로).
- 화면 2는 "목록"과 "상세"가 사실상 하나의 화면(리스트 뷰 ↔ 드릴다운 뷰)이라고 보고 두 번째 화면으로 합쳤다.

---

## ② 데이터: 무엇을 다루고, 배포 스냅샷엔 뭐가 들어가고 빠지는가

### 화면 1(로컬)이 다루는 전체 데이터
| 데이터 | 배포됨? |
|---|---|
| API 키/토큰 (`.env`) | ❌ 절대 제외 |
| 수집 원본(raw API 응답) | ❌ 제외 |
| 편집 중 임시 상태(초안) | ❌ 제외 |
| 프리셋(자주 쓰는 경쟁사/키워드) | ❌ 제외 |
| 비공개 메모(있다면) | ❌ 제외 — 배포 시 strip |
| **확정된 브리핑**(최종본) | ✅ 이것만 배포 |

### 화면 2(공개)에 들어가는 필드
```jsonc
{
  "date": "2026-07-14",
  "title": "...",
  "summary": "...",
  "top_importance": "high",
  "competitors": ["A사", "B사"],
  "items": [
    { "headline": "...", "note": "...", "importance": "high",
      "url": "...", "source": "...", "is_new": true }
  ]
}
```
원칙: **"화면 1에서 확정(승인)한 것만 화면 2에 나타난다."** 키·원본 데이터·초안은 애초에 이 파일에 담기지 않음.

---

## ③ 사용자가 쓰는 순서

```
화면 1 열기
  → 경쟁사·키워드 입력
  → "브리핑 생성" → 자동 수집 + 중요도 초안
  → 그 자리에서 편집(삭제/수정/중요도 조정)
  → "배포" 클릭 (공개 가능한 필드만 커밋·푸시)
      ↓ (몇 분 후)
화면 2에서 카드 확인 → 클릭 → 상세 확인
```

---

## ④ 무엇부터 만들지 (우선순위)

| 순서 | 작업 | 이유 |
|:---:|---|---|
| 1 | 데이터 스키마 확정 (예시 JSON 2~3개) | 두 화면 모두 이 스키마에 의존 |
| 2 | **화면 2 프로토타입** (mock 데이터로 카드+상세) | 외부 의존성 없이 바로 만들고 결과물을 눈으로 확인 |
| 3 | **화면 1** — 수동 입력으로 편집·중요도 조정까지 (자동 수집 제외) | 자동 수집 전에 편집 흐름·스키마를 먼저 검증 |
| 4 | 배포 파이프라인 (커밋·푸시, 공개 필드만 필터링) | 2·3 연결 |
| 5 | 자동 수집 연동 (뉴스/웹 API) | 외부 의존성이 가장 크므로 맨 뒤로 |
